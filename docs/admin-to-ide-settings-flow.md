# Admin 설정이 IDE에 반영되는 구조

## 1) 개요

CodePilot은 “관리자 정책이 실제 개발 화면까지 이어지는 구조”가 핵심입니다.  
즉, Admin에서 정한 운영 기준이 IDE 사용자 경험에 자동 반영됩니다.

쉽게 말해:

- Admin은 정책을 정하고
- IDE는 그 정책을 실행하며
- 사용자는 정책 범위 안에서 빠르게 개발합니다.

## 2) 반영 흐름(비기술 설명)

1. 관리자가 Admin에서 팀 정책을 설정합니다.  
2. 사용자가 IDE를 열면 조직 정책이 자동으로 동기화됩니다.  
3. 필수 정책은 즉시 적용되고, 권장 정책은 가이드 형태로 제공됩니다.  
4. 사용자는 허용 범위 안에서 개인 선호를 조정할 수 있습니다.  
5. 조직 정책이 바뀌면 IDE도 최신 정책으로 맞춰집니다.

## 3) 어떤 정책이 IDE에 반영되는가

### 1. AI 모델 정책

- 어떤 모델을 기본으로 사용할지
- 어떤 모델을 사용 금지/허용할지
- 조직 기준의 모델 운영 방향(품질 우선, 비용 우선 등)

**IDE에서 보이는 변화**
- 사용 가능한 모델 목록과 기본 선택 기준이 조직 정책대로 정렬됩니다.

### 2. 자동화 정책

- 자동 실행을 어디까지 허용할지
- 실패 시 재시도 기준을 어떻게 둘지
- 보수적 운영/공격적 자동화 운영 중 어떤 모드로 갈지

**IDE에서 보이는 변화**
- 사용자의 작업 방식이 팀 운영 원칙과 일치하게 맞춰집니다.

### 3. 빌드/검증 정책

- 팀 공통 검증 절차를 표준으로 반영
- 품질 점검 단계를 개발 흐름에 포함

**IDE에서 보이는 변화**
- 검증 누락이 줄고, 결과물 품질 기준이 일정해집니다.

### 4. 보안 정책

- 위험 작업 제한
- 민감 대상 보호
- 제외 대상 지정

**IDE에서 보이는 변화**
- 작업 속도는 유지하면서 보안 리스크를 낮출 수 있습니다.

### 5. 지식/규칙 정책

- 조직 문서 기반 지식 활용
- 팀 코딩 규칙/아키텍처 원칙 반영

**IDE에서 보이는 변화**
- “우리 팀 방식”에 맞는 결과를 더 안정적으로 얻을 수 있습니다.

## 4) 강제 정책과 권장 정책의 차이

### 강제 정책

- 반드시 지켜야 하는 기준입니다.
- 사용자가 임의로 바꾸기 어렵습니다.

**언제 쓰나**
- 보안, 컴플라이언스, 필수 운영 기준

### 권장 정책

- 기본 권장값으로 제공되는 기준입니다.
- 업무 특성에 맞게 사용자가 조정할 수 있습니다.

**언제 쓰나**
- 팀 생산성 최적화, 실험/탐색 영역

## 5) 적용 시나리오

### 시나리오 A: 보안 우선 조직

- 위험 작업/민감 대상 정책은 강제로 적용
- 모델 및 자동화는 제한적으로 개방

**효과**
- 규정 준수 중심으로 안정적 도입 가능

### 시나리오 B: 속도 우선 제품팀

- 자동화/검증 정책을 적극 적용
- 팀 규칙을 권장 정책으로 시작해 단계적으로 강화

**효과**
- 릴리즈 속도 향상과 품질 기준 동시 확보

### 시나리오 C: 대규모 조직 확산

- 본사 표준 정책을 공통 적용
- 팀별로 일부 권장 정책만 유연 운영

**효과**
- 중앙 통제와 현장 유연성을 함께 달성

## 6) 요약

CodePilot의 강점은 단순 AI 기능이 아니라,  
**관리 정책이 실제 개발 행동으로 연결되는 운영형 구조**입니다.  
그래서 “도입은 쉬운데 운영이 어려운 AI”가 아니라,  
“도입 후에도 통제 가능한 AI 개발 환경”을 제공합니다.
