모든 설정 페이지는 같은 방식으로 동작합니다.

- **추가** 버튼으로 항목을 등록하고, 항목마다 **수정 · 삭제** 가 가능합니다.
- 각 페이지에는 **추천 프리셋**이 준비되어 있어 클릭 한 번으로 바로 적용할 수 있습니다.
- 항목마다 **필수 / 권장** 강도를 지정합니다. 자세한 내용은 [정책 운영 가이드](/admin/policies)를 참고하세요.

## AI 모델

![AI 모델 설정](/images/ADMIN_AI_MODEL.png)

조직에서 사용할 AI 모델을 등록하고 승인합니다.

| 구분 | 설명 |
|------|------|
| **내장 모델** | 로컬 모델 서버 연동. 켜고 끄기만 하면 됩니다 |
| **지원 모델** | 주요 제공사별 모델 목록. 모델 또는 그룹 단위로 활성화 |
| **커스텀 모델** | 직접 등록하는 모델 (엔드포인트, 인증 방식, 파라미터 지정) |

지원하는 연동 유형은 공식 API, 오픈모델 API, 클라우드(Vertex AI · Azure OpenAI · AWS Bedrock), 자체 서버(OpenAI 호환 엔드포인트)입니다.

<Tip>
**모델 화이트리스트** — 조직이 승인한 모델만 IDE의 모델 선택 목록에 표시됩니다. 개인의 취향이 아닌 **회사의 비용·보안 기준**에 맞는 모델 전략을 운영할 수 있습니다.
</Tip>

## MCP 서버

![MCP 설정](/images/ADMIN_MCP.png)

GitHub, Jira 등 사내에서 승인한 외부 도구만 연동하도록 통제합니다.

- 서버 이름과 URL을 등록하면 조직 전체 IDE에 자동 배포됩니다.
- 인증은 API 키 또는 OAuth를 지원합니다.
- **연결 테스트**와 **도구 목록 가져오기**로 정상 동작을 확인할 수 있습니다.

<Tip>
확장성은 유지하되, 검증되지 않은 외부 도구 연결로 인한 **데이터 유출을 방지**합니다.
</Tip>

## RAG (사내 지식)

![RAG 설정](/images/ADMIN_RAG.png)

사내 위키, API 문서, 정책서를 업로드하면 AI가 이를 참고해 답변합니다.

| 작업 | 설명 |
|------|------|
| RAG 소스 생성 | 이름·설명과 함께 문서를 업로드 |
| 문서 추가 | 기존 소스에 문서를 더 올리기 |
| 리인덱싱 | 문서 변경 후 검색 색인을 최신화 |
| 검색 설정 | 유사도 임계값과 검색 결과 개수 조정 |

지원 형식: `.txt` `.md` `.pdf` `.docx` `.json` `.csv`

<Tip>
팀 공통 문서를 AI가 참고하여 **"우리 회사 맥락을 아는"** 답변을 제공합니다.
</Tip>

## Skills (개발 규칙)

![Skills 설정](/images/ADMIN_SKILLS.png)

AI가 코드를 작성할 때 따라야 할 팀 규칙을 Markdown으로 등록합니다.

```
- 변수명은 camelCase를 사용한다
- JPA Entity는 FetchType.LAZY를 기본으로 한다
- API 응답은 Result 래퍼 객체로 감싼다
```

카테고리는 버전 관리, 코딩 스타일, 프로젝트 아키텍처, 의존성 정책, DB, 커스텀으로 나뉩니다.

| 종류 | 적용 방식 |
|------|----------|
| **규칙(Rule)** | 항상 AI 프롬프트에 포함 |
| **스킬(Skill)** | 필요한 상황에만 불러와 사용 |

<Tip>
개발자가 규칙을 일일이 외우지 않아도, AI가 생성 단계에서 **팀 표준을 준수**합니다.
</Tip>

## 빌드/테스트 설정

![빌드/테스트 설정](/images/ADMIN_BUILD.png)

코드 생성 후 자동으로 실행할 검증 명령과 포맷터 명령을 등록합니다. 언어별로 다르게 지정할 수 있습니다.

```
검증:   tsc --noEmit
포맷터: prettier --write .
```

검증에 실패하면 AI가 에러를 분석해 스스로 수정을 시도합니다.

## Hot Load

![Hot Load 설정](/images/ADMIN_HOTLOAD.png)

키워드를 등록해두면, 팀원이 채팅에 그 키워드가 담긴 문장을 입력할 때 미리 정의한 명령이 자동 실행됩니다.

| 항목 | 설명 |
|------|------|
| 트리거 키워드 | 쉼표로 구분해 여러 개 등록 |
| 명령어 | 실행할 셸 명령 |
| 완료 조건 | 종료 코드, 출력 문자열 포함 여부, 파일 존재 여부 |
| 재시도 | 실패 시 재시도 횟수 |
| 실패 시 동작 | 중지하거나, 에러를 AI에게 전달해 해결 시도 |

<Tip>
팀의 반복 업무 노하우를 **자동화 규칙으로 자산화**합니다.
</Tip>

## Hooks

작업 흐름의 특정 시점에 셸 명령을 연결합니다. 감사 로그 기록이나 사내 시스템 연동에 활용합니다.

{/* SCREENSHOT: Admin Hooks 페이지 — 등록된 훅 목록과 이벤트/명령어/도구 매칭/타임아웃 입력 폼 */}
![Hooks 설정](/images/ADMIN_HOOKS.png)

| 이벤트 | 시점 |
|--------|------|
| 도구 실행 전 | AI가 도구를 실행하기 직전 |
| 도구 실행 후 | 도구 실행이 끝난 뒤 |
| 도구 실패 시 | 도구 실행이 실패했을 때 |
| LLM 호출 전 | AI에게 요청을 보내기 직전 |
| 세션 시작 시 | 새 대화를 시작할 때 |

특정 도구에만 반응하도록 매칭 조건을 지정할 수 있고, 동기/비동기 실행과 타임아웃을 설정합니다.

## Agents

전사 공통으로 배포할 서브에이전트를 등록합니다. 팀원은 IDE 채팅에서 **`@에이전트이름`** 으로 호출합니다.

{/* SCREENSHOT: Admin Agents 페이지 — 등록된 에이전트 목록(@이름, 허용 도구 수, 읽기전용 여부)과 입력 폼 */}
![Agents 설정](/images/ADMIN_AGENTS.png)

| 항목 | 설명 |
|------|------|
| 이름 | `@` 뒤에 붙일 호출명 |
| 설명 | 이 에이전트가 하는 일 |
| 시스템 프롬프트 | 역할과 지침 |
| 허용 도구 | 사용 가능한 도구 범위 |
| 읽기 전용 | 파일 수정·삭제 도구 차단 |

<Tip>
"보안 리뷰 담당", "테스트 작성 담당"처럼 역할을 나눠 배포하면, 팀원 모두가 **같은 기준의 검토**를 받을 수 있습니다.
</Tip>

## 보안 규칙

![보안 규칙 설정](/images/ADMIN_SECURITY.png)

위험한 명령 실행과 민감 파일 접근을 차단합니다.

| 유형 | 설명 | 예시 |
|------|------|------|
| **차단 명령어** | AI가 실행할 수 없는 명령 | `rm -rf /` |
| **보호 파일** | 읽기·수정을 막을 파일 | `.env*`, `credentials.json` |
| **파일 은닉** | AI가 존재조차 알 수 없게 감출 파일 | `config/production.json` |

## 제외 패턴

![제외 패턴 설정](/images/ADMIN_PATTERN.png)

프로젝트 인덱싱, 파일 검색, 컨텍스트 수집에서 제외할 경로를 glob 패턴으로 등록합니다. `.gitignore` 는 자동으로 반영됩니다.

```
.env*
node_modules/
dist/
```

<Tip>
보안팀과 개발팀의 갈등 없이, **시스템 레벨에서 안전 장치**를 마련합니다.
</Tip>
